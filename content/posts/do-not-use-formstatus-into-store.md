Form! 어떻게 보면 프론트엔드의 꽃이다.

로그인이나 회원가입처럼 대부분의 서비스에는 Form이 있다. 그래서 처음에는 Form을 만드는 게 어렵지 않을 거라고 생각했다. input만 만들면 되는 거 아닌가? (하하)

그런데 Form이 점점 복잡해지기 시작한다면? 필드가 늘어나고, 단순 submit form이 아니라 편집기에서 사용하려면 입력한 값도 어딘가에 저장해야 했다. 그럼 전역 store를 사용하면 되겠다고 생각했다.

> 폼 값도 결국 상태니까 똑같이 전역 스토어로 관리하면 되겠지?

처음에는 합리적인 선택처럼 보였지만... validation이 들어가고, 에러를 표시하고, 제출 가능 여부까지 관리해야 하면서 상황이 달라졌다. store에는 입력 값뿐 아니라 validation 상태와 에러 상태까지 쌓여가기 시작했다.

이 시점부턴 Form을 애초에 내가 관리를 한다는 발상 부터가 잘못되었다고 생각했고 Form의 validation을 어떻게 관리하는지 찾아보다가 React Hook Form (업계 표준!)을 알게 되었고 결국 도입하게 됐다.

'폼 설계 삽질기' 시리즈에서는 store 기반으로 Form을 관리하면서 겪었던 시행착오와 RHF을 도입하며 Form 구조를 어떻게 개선해 나갔는지 정리해보려고 한다.

## Form을 전역 스토어로 관리하기?

전역 스토어로 폼을 관리하는 구조는 이렇게 잡았다.

```ts
type FormStore<T> = {
  form: Partial<T>;
  patchForm: (fields: Partial<T>) => void;
};
```

페이지가 서버 데이터를 받아서 스토어에 넣고 입력 컴포넌트는 `patchForm()`으로 값을 고친다. submit 시엔 스토어 값을 그대로 서버로 보내면 끝인 형태였다.

이때는 form이 값을 넣거나 빼는 역할만 했기 때문에 별 문제 없다고 생각했다. 하지만 기획의 요구사항이 주어졌다. "이름이 입력 안되면 저장 안되게 해주세요!"

처음에는 그냥 어딘가에 조건을 정리해두고 그 조건 안맞을때 에러 반환하는 함수로 처리할 수 있겠다고 생각했다.
하지만 필수값 검증을 하려면 필드별 에러 상태, 데이터 dirty 여부 (전체 중 누굴 수정했는지), 배열 필드라면 몇 번째 값이 잘못되었는지 등등 생각보다 고려할게 많았다.

이걸 전역 스토어 구조에 넣으려면 결국 `errors`, `touched`, `dirtyFields` 같은 필드를 스토어에 추가로 만들어야 하 는 데...

스토어가 무엇인가? 스토어는 selector 등으로 범위를 세심하게 조절하지 않으면 값이 하나라도 바뀌었을 때 구독자 전체가 이 사실을 알게된다. `form` 객체 하나를 통째로 스토어에 두는 구조라면 필드 하나가 바뀔 때마다 이 객체 전체가 새로 만들어지고 그럼 그걸 구독하는 컴포넌트들이 함께 리렌더링 대상이 된다. form은 반대로 값이 바뀌어도 전체 화면이 그 변화를 알 필요는 없다.

> store -> 널리 공유되는 상태 / form -> 좁은 범위에서 자주 바뀌는 상태

그러니까 검증할 상태가 늘어날 수록 리렌더링은 더 잦게 무겁게 작동할 것이다.

### Controlled과 Uncontrolled

Controlled는 입력값을 React state로 들고 있다가 `onChange`마다 리렌더링하는 방식이다(내가 만든 스토어가 바로 이 방식을 채택). Uncontrolled는 DOM이 값을 들고 있다가 필요할 때만 읽어오는 방식이다(rhf이 이 방식).

## 구원타자 React Hook Form + Zod (검증된 라이브러리를 사용하자)

form을 직접 관리할게 아니라면 라이브러리를 선택해야했는데 솔직히 라이브러리를 고르는데 오래 걸리지는 않았다. 업계 표준이라고 불리는 React Hook Form(앞으론 RHF라고 부르겠다)이 있었기 때문이다.

솔직히 RHF의 내부 구현 원리에 대해서는 파악하고 도입한것은 아니지만 목표가 된 검증을 붙이기 쉬운 라이브러리라서 선택한게 크다. 바로 zod가 그 검증 라이브러리인데 zod는 폼 전체를 하나의 스키마로 파싱 -> 검증 결과를 필드 단위의 에러 객체로 변환해 RHF에 전달한다. 저장은 rhf가, 유효성 검사는 zod가 하는 구조다.

```text
[서버 데이터] -> useForm(defaultValues)
                     │
              사용자 입력 (ref 기반으로 필요 시에만 리렌더)
                     │
              submit 시 zodResolver로 검증 (커스텀)
                     │
              실패 -> 필드별 error 매핑 -> 화면에 각 필드별 표시 가능!
              성공 -> 서버로 전송
```

### 상태 관리와 폼 관리

|             | 전역 상태                          | 폼 상태                 |
| ----------- | ---------------------------------- | ----------------------- |
| 목적        | 여러 컴포넌트가 같은 데이터를 공유 | 하나의 입력 흐름을 추적 |
| 변경 빈도   | 상대적으로 낮음                    | 키 입력마다 발생        |
| 필요한 정보 | 값 자체                            | 값 + 유효성 여부        |
| 구독 범위   | 넓게 산개되어있음                  | 해당 필드나 form        |

검증된 방식인 라이브러리를 사용하기로 결정하고나니 엔티티별로 만들어둔 form store 13개랑 검증 store는 이만 작별할 수 있었다. (생 노가다...)

## Form 사용하면 항상 라이브러리를 도입해야할까?

모든 라이브러리 도입이 그렇듯 트레이드오프가 당연히 있다.

- **필드가 적고 검증이 없거나 아주 단순하다면** -> 전역 스토어나 심지어는 로컬 `useState`로도 충분하다.
- **폼 외부에서도 변경된 값을 알아야한다면** -> 전역 스토어로 관리를 하고 selector 등으로 스코프를 좁혀 리렌더가 덜일어나도록 사용하자.
- **필드가 많고, 배열이나 중첩 구조가 있고, 필드별로 에러를 관리해야한다면** -> 복잡한 요구사항을 이미 풀어놓은 라이브러리가 있나 찾아보고 검증된 방법을 도입하자!

## 배운점

요구사항이 복잡해질수록 직접 구현하는 것이 꼭 좋은 설계는 아닌 것 같다.항상 비슷한 고민을 했던 선구자 개발자/커뮤니티가 있었을것임을 잊지말자. 라이브러리를 도입하는 것이 능사는 아니지만 요구사항의 스코프와 복잡도를 고려해 이 문제를 어떻게 풀지를 고려하는 것보다는 이 문제를 이미 풀어낸 라이브러리가 있는지 찾아보는 것이 더 빠르고 효율적일 수 있다.
