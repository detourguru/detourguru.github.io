개발 중에 특정 화면에서 가끔 아래와 같은 콘솔 에러가 발생했다.

```text
TypeError: Cannot read properties of undefined (reading 'name')
```

```tsx
function GameInfo({ game }) {
  return <div>{game.name}</div>;
}
```

확인해 보니 대부분의 경우에는 정상적으로 동작했고 간헐적으로 `game`이 `undefined`로 들어오고 있었다.

Suspense query를 사용하다보니 비슷한 로그가 찍히는 경우가 종종 있었고 그래서 _당연히_ 렌더링 타이밍 문제일거라고 생각해 그냥 화면이 깨지지 않게만 수정했다.

```tsx
function GameInfo({ game }) {
  try {
    return <div>{game.name}</div>;
  } catch {
    return null;
  }
}
```

콘솔 에러는 사라졌고 화면도 정상적으로 보였다. 새로고침 몇 번 해보고 재현되지 않는 것을 확인한 뒤 일단 해결했다고 판단했다.

## 3개월 뒤...

사내 테스트 중 신규 가입 사용자에게 문의가 들어왔다.

> "게임 정보 화면이 비어 있어요."

당연히 `game` 값부터 확인했다. 아니나 다를까 undefined였다.

```tsx
console.log(game);
// undefined
```

try-catch에 숨겨져있던 에러가 다시 등장한 것이다.

```text
TypeError: Cannot read properties of undefined (reading 'name')
```

그런데 이번에는 새로고침 몇 번에 해결되지 않았다. 원인을 파악해야했다.

## 엣지케이스를 놓치다

데이터 흐름을 다시 따라가보았다.

> 사용자가 게임 화면 진입 > `useGameQuery()` 호출 > API 응답을 store에 저장 > `GameInfo`가 store의 game 데이터를 사용해 렌더링

문제는 신규 사용자였다. 신규 사용자는 아직 게임 데이터가 없는 상태였고 API가 404를 반환했다.

하지만 `fetch`를 사용하고있어 에러로 throw되지 않았고 데이터가 없는 상태가 정상 흐름처럼 처리되며 결국 `undefined`가 컴포넌트까지 전달된 것이다.

## 이렇게 수정했습니다.

먼저 에러를 숨기던 `try-catch`를 제거한 다음 데이터가 없는 상황을 하나의 정상적인 상태로 보고 처리했다.

```ts
const { data: game } = useGameQuery({
  gameId,
  select: (res) => (res.status === 404 ? null : res.data),
});
```

테스트 코드도 다시 확인했다. 테스트도 정상적인 사용자 흐름만 기준으로 작성되어 있었다.

기존 시나리오에는 게임 데이터가 존재하는 사용자와 정상적으로 API 응답을 받는 상황만 가정되어있었지만

아직 데이터가 없는 사용자나 API가 에러 응답을 반환하는 상황에 대한 시나리오는 없었다.

솔직히 이 테스트코드만 작성되어 있었더라도 충분히 미연에 방지할 수 있는 문제였다.

이후 새로운 화면을 만들 때도 정상 데이터만 생각하지 않고 "데이터가 없을 때 어떻게 보이는가"를 먼저 확인하게 됐다.

- 데이터가 정상적인 상황
- 에러 응답이 온 상황
- 초기 진입 상황

최소 이 세가지는 항상 고려하거나 설계적으로 공통 컴포넌트를 미리 주입 받을 수 있도록 고려해서 화면을 작업하게 되었다.

## 에러 모니터링 추가

추가로 미뤄두었던 Sentry도 도입했다.

기존엔 매번 개발자가 console.log를 찍어서 확인할 수는 없으니 한계가 있었지만
Sentry 도입 후에는 어떤 화면에서 어떤 에러가 발생했는지 실제 사용자에게 영향이 있었는지 먼저 확인할 수 있게 됐다.

## 배운점

그렇게 어려운 문제도 아니었는데 원인을 확인하지 않은 채 추측으로 그러려니 하고 해결했다고 생각했더니 유저 리포트로 돌아온 사례였다.

이번 경험 이후 작은 에러라도 바로 숨기기보다 먼저 "어떤 조건에서 발생하는가"를 먼저 확인하게 되었다.

특히 정상 데이터가 있는 상황만 생각하지 않고 데이터가 없는 상황과 실패하는 상황까지 같이 설계하는 습관을 가지게 됐다.
