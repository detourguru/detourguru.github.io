하나의 컴포넌트가 동적 폼처럼 여러 타입의 필드를 전부 처리하게 만들면 처음 만들 떈 정말 편하다. 그런데 기능이 하나둘 늘다 보면 어느 순간 손대기도 무섭고 어디부터 살펴야할지도 모를 파일이 되어 있다.

내가 만든 `DynamicForm`이라는 컴포넌트가 딱 그랬다. 아이템, 장소, 이벤트처럼 서로 다른 엔티티를 하나의 폼 시스템으로 편집해야 하는 컴포넌트였고 정해진 타입으로 prop을 넘기면 자동으로 폼을 그려주는 컴포넌트였다.

```tsx
<DynamicForm
  parentId="items[]"
  fields={[
    { id: "name", type: "input", title: "이름" },
    { id: "type", type: "select", title: "타입" },
    { id: "icon", type: "media", title: "아이콘" },
  ]}
/>
```

사용하는 쪽 인터페이스는 간단했지만 점점 더 다양한 엔티티와 분기를 추가하다보니 내부는 시간이 지나며 580줄까지 불어났다. 타입 정의, 값 탐색 헬퍼, 8개 필드 렌더링, Row 컴포넌트, Drag & Drop, React Hook Form 연동이 전부 한 파일에 있었다.

### 거대 컴포넌트의 단점

일단 컴포넌트 내부가 복잡할때의 단점은 이랬다.

- **수정 범위를 파악하기 어려웠다.** 필드 타입 하나만 고치려 해도 180줄짜리 삼항 체인을 처음부터 끝까지 따라가야 했다.
- **테스트를 추가할 수 없었다.** 컴포넌트 하나가 너무 많은 책임을 가지고 있다보니 어디서부터 어디까지를 테스트해야 할지조차 알기 어려웠다.
- **입력할 때마다 전체가 다시 렌더링됐다.** 예를들어 필드 30개짜리 폼에서 한 글자를 입력하면 30개 Row가 전부 다시 그려졌다.

## 그럼 어떻게 쪼개야 할까?

내가 가장 먼저 떼어낸 건 타입과 순수 헬퍼였다. 일단 독립적으로 떼어내기도 쉬웠고 영향 범위가 좁았기 때문이다.

그리고 그 와중에 개선도 할 수 있었다. 중첩된 값을 읽어오는 `getNestedValue()`를 직접 구현해 쓰고 있었는데, 테스트 시나리오를 작성하다 보니 닷 노테이션 말고도 배열 인덱스나 대괄호 표기 같은 경로도 처리해야 한다는 게 드러났다. 이미 검증된 방식인 lodash를 사용하도록 변경했다. 더 좋은건 이미 프로젝트 내에 다른 경로 검증 커스텀 로직들이 있었어서 다같이 lodash로 변경을 할 수 있었다. (불필요한 라이브러리를 늘린다고 좋은게 아니니까)

```ts
import { get } from "lodash";

export function getNestedValue(obj, path) {
  return get(obj, path);
}
```

이 과정에서 `isFieldVisible()`이 조건 필드가 `undefined`일 때 잘못 표시되는 버그도 발견했다. 리팩토링용으로 짠 테스트가 기존 버그를 잡아낸 셈이다. 리팩토링이 이래서 재미있다. 모르던 숨은 버그도 찾게 되니까!

## 삼항 체인 대신 레지스트리 패턴

가장 복잡했던 건 필드 렌더링이었다. 원래는 이런 식으로 타입마다 삼항 체인을 이어붙이고 있었다.

```tsx
field.type === 'input'
  ? <Input />
  : field.type === 'select'
  ? <Select />
  : field.type === 'media'
  ? <Media />
  : ...
```

새 필드 타입이 추가될 때마다 이 체인을 계속 수정해야 하는 구조였다. 이걸 타입별 컴포넌트를 매핑해두는 레지스트리 패턴으로 바꿨다.

```tsx
export const fieldRenderers = {
  input: InputRenderer,
  select: SelectRenderer,
  media: MediaRenderer,
  badge: BadgeRenderer,
};

const Renderer = fieldRenderers[field.type] ?? fieldRenderers.input;

return <Renderer {...props} />;
```

이렇게 바꾸면 새 필드 타입을 추가할 때 기존 코드는 건드릴 필요 없이 레지스트리에 한 줄만 추가하면 된다. 여러 타입을 분기 처리하는 코드가 계속 늘어나는 상황이라면, 삼항이나 switch 체인보다 레지스트리 패턴이 확장에 유리하다!

## 배운 점

거대 컴포넌트는 꼭 쪼개야할까? 그렇진 않다. 문제는 길이가 아니다.

- 책임이 뒤섞여 있는가? -> 한 부분을 고칠 때 어디까지 영향이 갈지 코드만 보고 알 수 없다면 책임이 뒤섞여있다는 말이다.
- 새로운 케이스 추가 시 기존 코드 수정이 불가피한가? -> 그렇다면 분기 체인을 레지스트리 패턴으로 바꾸는 걸 고려할 시점이다.
- 독립적 테스트가 가능한가? -> 안 된다면 순수 로직부터 먼저 떼어내는 게 안전하다.

쪼갠다고 다 능사는 아니다. 안전하게 수정할 수 있는 구조를 만드는 게 목적이라는 걸 기준으로 삼으면 어디부터 쪼갤지 판단하기가 한결 쉬워진다.
