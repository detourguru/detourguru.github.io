## 580줄 `DynamicForm`을 5개 모듈로 분해한 리팩토링

내 마음 속의 리팩토링 1순위였던 컴포넌트를 어느날 딱 열어봤는데 스크롤이 정말이지 끝나질 않았다.

쓴 사람인 나는 알아볼 수 있었고 대략 이 기능이 어디쯤에 있더라~ 알 수 있었지만 다른 사람은 아마 보기 꽤나 골치였을 것이다.

그도 그럴 것이 타입 정의부터 헬퍼 함수, 필드 렌더링, DnD 까지 모두 하나의 컴포넌트 안에 들어 있었다. (변명을 좀 해보자면 개발 초기에 빠르게 쳐내려고 별도로 분리를 안했다.)

기능은 잘 동작했지만 새로운 기능을 추가하거나 버그를 수정할 때마다 어디를 건드려야 할지 찾는 것부터 어려운 코드가 되어 버린 것이다.

이번 리팩토링의 목표는 단순히 파일을 나누는 것이 아니었다.

- **600줄 짜리 단일 컴포넌트를 책임별로 분리**
- **테스트 추가 (복잡한 컴포넌트인데도 테스트 하나 없었다니!)**
- **불필요한 리렌더 제거**
- **안전하게 수정할 수 있는 구조 만들기**

결과적으로 `DynamicForm.tsx`는 약 100줄만 남았고, 테스트도 없던 거대한 컴포넌트에서 역할이 명확한 모듈들로 나눌 수 있었다.

## DynamicForm은 어떤 컴포넌트였나

`DynamicForm` 이 무슨 컴포넌트냐하면 프로젝트 아주 초기에 처음만들어진 터줏대감같은 Form이라고 할 수 있겠다.

게임 에디터에서는 아이템, 장소, 이벤트처럼 서로 다른 엔티티를 하나의 폼 시스템으로 편집해야 했다.

예를 들어 아이템은 이름, 타입, 아이콘을 가지지만 이벤트는 조건과 보상 목록을 가진다.

이를 `FieldDefinition[]`으로 정의하면 자동으로 폼을 생성하는 것이 `DynamicForm`의 역할이었다.

```tsx
<DynamicForm
  parentId="items[]"
  fields={[
    { id: "name", type: "input", title: "이름" },
    { id: "type", type: "select", title: "타입" },
    { id: "icon", type: "media", title: "아이콘" },
  ]}
/>
```

사용하는 쪽은 간단했지만 점차 까다로워지는 조건들 때문에 내부 구현은 시간이 지나면서 점점 커졌다.

## 리팩토링 전 상태

하나의 파일 안에는 다음 역할들이 모두 섞여 있었다.

- 타입 정의
- 값 탐색 및 조건 계산 헬퍼
- 8가지 필드 렌더링
- Row 컴포넌트
- Drag & Drop 연결
- React Hook Form 연동

문제는 단순히 "파일이 길다"는 것이 아니었다.

### 수정 범위를 파악하기 어려웠다

특정 필드 타입 하나를 수정하려 해도 180줄 가까운 삼항 체인을 따라가야 했다.

새 필드 타입을 추가할 때마다 기존 코드를 계속 수정할 수밖에 없는 구조라 확장성도 좋지 않았다.

### 테스트를 추가하기 어려웠다

컴포넌트 하나가 너무 많은 책임을 가지고 있어서 단위 테스트를 작성할 수 없었다.

셋업 비용도 크지만 어디서부터 어디까지 뭘 테스트해야할지 알 수 없는 상태가 되어 방치가 계속 되었다.

### 입력할 때마다 모든 Row가 다시 렌더링됐다

`DynamicForm`이 다시 렌더링되면 내부 Row들도 모두 다시 렌더링됐다.

아이템이 30개인 폼에서는 한 글자를 입력할 때마다 30개의 Row가 모두 다시 그려졌다.

## 리팩토링 전략

한 번에 갈아엎지 않고 타입 → 헬퍼 → 렌더링 → Row → 훅 → 통합 테스트 순서로 진행했다.

## 타입 / 순수 로직부터 분리

가장 먼저 타입과 순수 헬퍼를 분리했다. 제일 간단하기도 하고 금방 할 수 있는 부분이니까.

그런데 그 과정에서 생각보다 많은 개선을 할 수 있었다.

예를 들어 `getNestedValue()`를 처음에는 직접 구현했다.

```ts
path.split('.').reduce(...)
```

테스트 시나리오 부터 작성하다보니 단순히 닷노테이션말고도 배열 인덱스나 대괄호 표기 같은 다양한 경로를 처리해야 한다는 점을 발견했다.
분명 더 좋은 검증된 방식이 있을 것 같았고 서치 끝에 lodash를 사용하게 변경했다. (더 좋은건 이미 포함되어있던 라이브러리라는 것이다)

```ts
import { get } from "lodash";

export function getNestedValue(obj, path) {
  return get(obj, path);
}
```

### 버그 발견

`isFieldVisible()` 테스트를 작성하다가 조건 필드가 `undefined`일 때 잘못 표시되는 버그도 발견했다.

리팩토링을 위해 작성한 테스트가 기존 버그를 찾아낸 첫 경험이었다!

## 삼항 체인을 레지스트리 패턴으로 변경

가장 복잡했던 부분은 필드 렌더링이었다.

기존에는 모든 타입을 하나의 삼항 체인으로 처리했다.

```tsx
field.type === 'input'
  ? <Input />
  : field.type === 'select'
  ? <Select />
  : field.type === 'media'
  ? <Media />
  : ...
```

새로운 필드 타입이 추가될 때마다 기존 코드를 수정해야 했다.

그래서 사용하는 쪽에서 타입에 맞는 Renderer만 선택하면 되도록 Renderer 레지스트리로 변경했다.

```tsx
export const fieldRenderers = {
  input: InputRenderer,
  select: SelectRenderer,
  media: MediaRenderer,
  badge: BadgeRenderer,
};
```

```tsx
const Renderer = fieldRenderers[field.type] ?? fieldRenderers.input;

return <Renderer {...props} />;
```

덕분에 새로운 필드를 추가할 때는 코드엔 손 안대고 Renderer에만 추가해주면 되어서 보다 더 확장성 있는 구조로 변하게 되었다.

사실 구현 과정에서 Renderer마다 Props가 조금씩 달라 타입 설계에 꽤 시간을 썼는데 결국 공통 Props를 기준으로 맞추고 필요한 값만 optional로 받는 방식으로 정리했다.

솔직히 완벽한 타입 설계는 아니었지만 현재 요구사항에서는 가장 단순한 구현이었다.

## Row를 분리하고 리렌더를 줄였다

각 Row를 `FormItemRow`라는 별도 컴포넌트로 분리하고 `React.memo`를 적용했다.

```tsx
const handleRemove = useCallback((index: number) => remove(index), [remove]);
```

이런식으로 함수 참조를 안정화한 뒤 확인해보니 이전에는 한 글자를 입력할 때마다 모든 Row가 다시 렌더링됐지만 이후에는 수정한 Row만 다시 렌더링되는 것을 확인할 수 있었다.

## 배운 점

가장 큰 변화는 파일 개수가 아니라 **안전하게 수정할 수 있는 기반이 생긴 것**이었다.

테스트를 추가하면서 숨어 있던 버그를 발견했고 책임을 분리하면서 컴포넌트 구조도 훨씬 이해하기 쉬워졌다.

무엇보다 `일단 동작하니까 건드리지 말자`의 대표주자였던 컴포넌트가 `누구나 이해하고 수정 가능한 컴포넌트`로 바뀌었다는 것이 이번 리팩토링의 가장 큰 성과였다.
