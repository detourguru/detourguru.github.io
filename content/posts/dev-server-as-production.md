사내테스트를 할 때의 일이다. 성능 리포트가 두 개 들어와 있었다.

> 에디터에서 저장이랑 조회가 느려요.

뭐 솔직히 이건 익히 알고있는 현상이었다. 데이터가 커서 느린거라 생각했지만 그래도 어딜 더 최적화해볼 수 있을지 확인해보니 전역 invalidate이 붙어있었다. 그래서 타깃 invalidate로 바꾸고 Suspense 워터폴을 prefetch로 병렬화했다. 그리고 노드 트리 쪽에도 불필요한 연산이 붙어있어서 이것도 최적화해줬다. 여전히 좀 느리긴했지만 저장 시간을 4초대에서 2초대로 개선할 수 있었다.

> 해외에서 접속하니까 멈춘 것처럼 스플래시 화면이 한참동안 떠있어요.

그런데 이건 처음보는 에러였다. 내가 외국으로 갈 수는 없으니 vpn으로 접속해서 비슷한 환경을 만든다음 테스트해보니 정말 느렸다! 에러 나서 화면이 아예 안뜨는게 아닌가 싶을정도로 말도 안되게 느렸다. 그런데 이상한건 첫 접속시에만 엄청나게 느리고 한번 화면이 뜨고나면 빨랐다는 것이다.

화면이 뜨고 나면 빠르다는건 코드 문제는 아닌 것 같았고 네트워크 패킷을 살펴보니 서버문제는 더더욱 아니었다. 그럼 대체 뭘까? 코드 문제가 아니라면 환경?

preview에서 확인했을 때 실제 로컬과 결과가 달랐던 경험이 있었기 때문에 `npm run preview`로 띄워서 테스트 해봤더니... 빨랐다! 그래서 서버에 들어가서 서빙을 무슨 명령어로 했나 확인해봤다. 결과는 충격적이었다.

```
$ pm2 show public_server
│ script path │ /usr/bin/npm │
│ script args │ run dev      │ // 여기!
│ uptime      │ 24D          │
```

사용자 트래픽이 Vite **개발 서버**로 서빙되고 있었다.

## 돌아간다고 운영할 수 있는건 아니다.

서버 셋업 초기 단계에 멋모르고 dev로 띄워서 확인했던게 화근이었다. 빌드 산출물을 따로 올리는 단계가 없으니 빠른 반영이 가능했고, 아직 배포되지 않은 단계의 우리 서비스에는 이게 편하고 잘 맞았던 것이다.

|                   | `npm run dev` (vite dev)       | `vite preview`                    | 정식 정적 서버 (nginx...) |
| ----------------- | ------------------------------ | --------------------------------- | ------------------------- |
| 서빙 대상         | 소스 원본 (native ESM, 미번들) | `dist/` 빌드 산출물               | `dist/` 빌드 산출물       |
| 목적              | 로컬 개발 중 즉시 반영         | 빌드 결과를 로컬에서 미리 확인    | 실제 서비스 트래픽 처리   |
| 압축(gzip/brotli) | 없음                           | 기본 미설정                       | 있음(설정 시)             |
| 개발용 통로       | HMR 웹소켓 등 노출             | 최소                              | 없음                      |
| 공식 입장         | 로컬 전용                      | 프로덕션 서버로 쓰지 말 것을 명시 | 프로덕션 용도             |

## 고치기 전에 계측부터 하기

서빙 방식을 바꿔야한다는건 자명했지만 그 전에 스플래시가 사라질 때 로딩 지표를 sentry로 보내는 계측 코드부터 먼저 배포했다.

```ts
// 스플래시 제거 직후 전송
Sentry.captureMessage("first_load_timing", {
  level: "info",
  extra: {
    splashDurationMs: Math.round(performance.now()),
    ttfbMs: navEntry ? Math.round(navEntry.responseStart) : undefined,
    scriptCount: scriptEntries.length,
    scriptTransferSizeBytes: scriptEntries.reduce(
      (s, e) => s + (e.transferSize || 0),
      0,
    ),
  },
});
```

서빙 방식을 변경하고나서 얼마나 개선되는지 궁금했기도 하고 이 현상을 리포트한 게 대표님이시다 보니 좀더 정확한 지표로 보고를 드리고 싶은 마음도 있었다.

### 실측 Before/After는 이랬다

|                  | Before (dev 서빙)                                    | After (정적 서빙) |
| ---------------- | ---------------------------------------------------- | ----------------- |
| 스플래시 시간    | **5,972ms** (첫 방문, 콜드) / 810ms (재방문, 근거리) | **128ms**         |
| 스크립트 요청 수 | **249개**                                            | **1개**           |
| 전송량           | 167KB ~ **8.8MB** (무압축)                           | 612KB (gzip)      |
| TTFB             | 9ms                                                  | 6ms               |

이전엔 (해외에서 테스트 시) 첫 방문 시 6초 걸려 로드됐지만 실제로는 167kb밖에 못받았었는데, 이후에는 8.8MB를 받아도 0.8초만에 로드가 끝났다. 스크립트 전송량이 많아서 오래 걸린 것이 아니었단 말이다!

## 왕복 횟수 × RTT의 함정

dev 서버는 번들을 만들지 않고 소스를 모듈 단위 그대로 서빙한다. 그런데 브라우저는 파일 하나를 받아서 열어봐야 그 안에 어떤 import가 더 있는지 알 수 있고 그걸 안 다음에야 다음 파일을 요청한다. 그래서 화면 하나 띄우는데 249번을 이렇게 하나씩 물어봐야 했던 것이다. 게다가 HTTP/1.1은 6개 동시 연결 제한이 있어서 249개 요청이 병렬로 안 나가고 순차적으로 왕복이 필요했다. 그러니까... 꼬리에 꼬리를 물고 돌아온 문제였다.

더 중요한 것은 이 리포트가 해외에서 테스트한 사용자에게서 들어왔다는 것이다. 왕복 한 번의 비용은 거리(RTT)에 비례한다. 사내 테스트는 같은 사무실 네트워크라 RTT가 몇 ms 수준이었고, 249번을 순차로 왕복해도 다 합쳐야 수백 ms였다. 그래서 사내에서는 속도 관련 리포트가 실제로 들어오지 않았고, 이게 정상적인 속도라고 착각할 수 밖에 없었다.

반면 해외에서는 RTT가 수백 ms 단위였다. 249번의 왕복에 그 RTT를 그대로 곱하면 초 단위, 심하면 수십 초까지 늘어난다. 왕복해야 하는 횟수는 국내에서든 해외에서든 똑같았지만 그 한 번의 비용이 달랐다. 내부 로직을 개선해서 조금 빨라진 에디터도 결국엔 이와 같은 이유로 여전히 느렸던 것이다.

로컬에서 dev/prod를 나란히 재본 수치는 이랬다.

| 시나리오 (로컬 비교)            | dev   | prod 빌드 | 배율   |
| ------------------------------- | ----- | --------- | ------ |
| 에디터 진입 메인스레드 블로킹   | 6.2s  | 1.78s     | ~3.5배 |
| 엔티티 조회 클릭 (코드 개선 후) | 1.65s | ~0.8s     | ~2배   |
| Suspense 스텝당 리렌더 갭       | 420ms | 110ms     | ~4배   |

dev 수치로 최적화 우선순위를 정하고 있었으니 사실상 삽질을 하고 있었던거다.

## 진짜 운영으로 바꿔주다

당연히 제일 먼저 한 일은 서빙 방식을 Preview로 바꿔준거였다. 아직 사내테스트가 진행중이었기 때문에 nginx로의 이관은 TODO로만 남겨놨다.

```bash
npm run build
pm2 delete public_server
pm2 start npm --name public_server -- run preview -- --port 3000 --host 0.0.0.0
pm2 save
```

사내테스트를 완료한 직후 nginx로 서버 설정을 변경했고 캐싱 설정도 넣어주었다. 당연히 훨씬 더 빨라졌다!

## 배운 점

물론 나도 변명을 해보자면 이런 서버 설정을 해본적도 없었기 때문에 처음에 좀 더 편하자고 개발 모드로 서빙을 해놓고도 까먹은 것에 대해서는 약간의 면죄부를 줄 수 있지 않을까 싶다... 게다가 아직 배포하는 서비스도 아니었고 대부분 사내에서만 테스트할 일이 있었기 때문에 이런 최적화나 설정은 할 일이 없었다.

처음엔 코드를 한참 들여다봤고 실제로 코드에 문제가 있어서 열심히 고쳤지만 결국 진짜 문제는 운영 서버가 npm run dev로 돌아가고 있었다는 사실이었다. 이번 일을 겪으면서 성능 문제라고 해서 꼭 코드가 문제가 아닐수도 있다는걸 배웠다.

또 하나 기억에 남는 건 Sentry였다. 붙인지 얼마 안 돼서 성능 계측을 달아봤는데 그 로그를 따라가면서 원인을 찾아가는 과정이 생각보다 정말 재밌기도 했고, 수치로 수정 전/후를 비교할 수 있으니 이게 우연찮게 빨라지고 느려진건지 원인을 수정해서 그렇게 된건지 파악할수 있다는 점에서 좋다고 생각했다. 이 이후로도 sentry를 이용해 계측을 달고 실측을 통해 원인을 도출해내게 되었다.
