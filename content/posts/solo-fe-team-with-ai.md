## 혼자 개발하면서 가장 어려웠던 것

> 신규 프로젝트라고만 했지 제가 혼자 다 결정해야 한다곤 말씀 안 하셨잖아요...

지금 맡고 있는 서비스는 게임 에디터를 비롯해 실시간 채팅 서비스인 인게임 화면, 그리고 플랫폼의 프론트엔드 전체다. 열심히 요구사항에 맞추어 작업하다 보니 어느새 라우트 43개, TS/TSX 약 15만 줄 규모가 됐고 그 사이의 크고 작은 결정도 전부 혼자 내려야 했다.

사수가 없는 환경에서 가장 부족했던 건 단순히 구현이 어렵다거나 내 코드를 봐줄 사람이 없다는 게 아니었다. **내 선택이 맞는 방향인지 검증받고 다른 관점을 얻을 기회가 부족하다는 점**이 가장 어려웠다.

기능 구현은 검색하고 문서를 찾아보면서 해결할 수 있었다.

하지만 판단을 해야할 순간에는 항상 고민이 생겼다.

- 이 로직을 컴포넌트로 분리하는 게 맞을까?
- Context로 해결할까, 상태 관리 라이브러리를 사용할까?
- 지금 구조가 나중에 변경하기 어려운 형태는 아닐까?

코딩에 정답은 없다지만 더 나은 방향은 분명히 있을텐데 그걸 알 수가 없으니 답답했다.

## AI를 코드 생성기로만 사용했을 때 생긴 문제

처음에는 AI를 디버거와 코드 생성 도구 정도로 사용했다.

> "이 에러 무슨 뜻이야?" "반복되는 코드 함수로 추출해줘."

요청한 범위 안에서 AI는 빠르게 답을 만들어줬지만 문제가 있었다. 내가 잘못된 전제를 가지고 질문하면 AI도 그 전제 위에서 코드를 만들어준다는 것이다.

그런데 어떻게 내가 잘못된 전제를 가지고있는지 알 수 있을까? (그걸 알면 이렇게 개발 안했을텐데) 그래서 AI를 개발 과정의 리뷰어로 활용하기 시작했다.

## AI에게 리뷰 역할 맡기기

현재는 작업을 크게 다음 순서로 진행한다.

1. 요구사항과 변경 범위를 정리한다.
2. handover 문서에 변경 내용과 트레이드오프를 작성한다.
3. AI에게 설계상 빠진 부분이나 요구사항을 벗어난 가정이 없는지 검토받는다.
4. 테스트 시나리오를 먼저 작성한다.
5. 구현한다.
6. 구현 후 다른 AI 세션으로 코드 리뷰를 한 번 더 받아 기존 코드와의 충돌이나 영향 범위를 확인한다.

이 과정에서 가장 효과를 본 습관은 구현 전에 테스트 시나리오를 먼저 작성하는 것이었다 (1인팀의 장점! 원하는 개발문화 자가 구축하기). 구현보다 먼저 요구사항을 검증하다 보니 놓친 조건이나 비약적인 가정을 발견하는 경우가 많았다.

이렇게 쌓인 테스트가 2600케이스 정도 된다. 해피패스는 필수고 버그가 발생할때마다 분기를 커버하는 회귀 테스트코드 우선 전략이다. 그러다보니 막상 Statements 커버리지는 30%가 채 안되는데 Branches는 75%를 커버하는 희한한 구조가 됐다. 처음에는 Statements 커버리지를 높이는걸 신경썼었는데 요구사항이 빠르게 바뀌는 환경이라 커버리지 수치보다 회귀 방어에 투자하는 게 낫다고 판단했다. 시나리오 설계는 사람이 하고 반복적인 케이스 작성은 AI가 분담하는 구조라 1인 팀에서도 이 정도 규모를 유지할 수 있었다.

### 테스트 시나리오 사전 점검의 장점

채팅 화면의 Unread Indicator 기능을 구현할 때였다.

처음에는 읽지 않은 메시지의 위치를 `index` 기준으로 저장하려고 했다.

```tsx
// index 기반으로 unread 시작점 저장
const unreadStartIndex = messages.length - unreadCount;
```

테스트 시나리오를 작성한 뒤 AI에게 리뷰를 요청하자 이런 질문을 받았다.

> 이전 메시지를 prepend하면 index는 어떻게 되는가?

실제로 로그를 확인해보니 이전 메시지를 prepend하면 기존 메시지들의 index가 모두 뒤로 밀렸다. 또 prepend가 발생하지 않아도 기존 기능대로 사용자가 처음 채팅방에 진입할 때 Greeting 메시지가 추가되면 기대했던 index가 달라질 수 있었다.

사용자가 처음 화면에 진입했을 때의 읽지 않은 메시지 위치를 유지하는 것이 요구사항이었는데 변동성이 있는 index 값은 요구사항을 만족시키기 위한 기준으로 적절하지 않았다.

반면 messageId는 prepend나 Greeting 메시지가 추가되어도 변하지 않는 값이었다. 그래서 messageId를 기준 값으로 사용하게 설계를 변경했다.

```tsx
// 불변값인 messageId 기준으로 저장
const unreadStartMessageId = messages[messages.length - unreadCount]?.id;
```

| 기준              | index     | messageId                            |
| ----------------- | --------- | ------------------------------------ |
| prepend 시 안정성 | 값이 밀림 | 불변                                 |
| 구현 난이도       | 낮음      | id로 위치를 다시 찾는 lookup 필요    |
| 최종 선택         | -         | 요구사항(최초 진입 시점 유지)에 부합 |

lookup 비용이 추가되긴 하지만 unread indicator 위치는 최초 진입 시 한 번만 계산하면 되는 값이었다. 그래서 이 정도 비용은 감수하고 요구사항의 안정성을 우선순위로 두는 게 맞다고 판단했다.

이후에는 이전 메시지를 추가로 불러오거나 Greeting 메시지가 생성되어도 Unread Indicator가 안정적으로 유지될 수 있었다.

```ts
// messageId로 index 찾기
// 첫 진입 시 messages는 최대 100개여서 크기가 크지 않아 가능했다.
const unreadIndex = messages.findIndex(
  (message) => message.id === unreadStartMessageId,
);
```

## 작업은 맡기고, 판단은 직접 하기

반복적인 구현 작업에서는 특히 AI를 적극적으로 활용하고있는데, 예를 들어 기존 패턴이 있는 화면을 추가하거나 비슷한 컴포넌트를 만드는 작업은 기존 코드를 참고해서 구현하도록 요청한다.

디자인 시스템이나 컴포넌트 규칙도 함께 전달하기 때문에 일관성 있는 결과물을 얻는 데 도움이 된다. 또 구현 전에 작성했던 handover와 트레이드오프를 기준으로 실제 구현이 요구사항을 벗어나지 않았는지도 다시 확인한다.
다만 결과물을 그대로 사용하는 것은 아니다. 가끔 발생하는 할루시네이션을 방지하기위해 기존 코드에서 확인된 내용인지, 일반적인 패턴인지 구분해서 답변하도록 요청했다.

## 이해도 게이트

여러 번의 검증을 거치더라도 가장 중요하게 생각하는 원칙이 하나 있다.

> **내가 이해하지 못한 코드는 머지하지 않는다.**

AI가 작성한 코드를 그대로 반영하면 이후 문제가 발생했을 때 원인을 추적하기 어려워지고 유지보수 비용도 커질 수 있기 때문이다. 그래서 커밋 전에는 이해도 게이트를 통해 항상 아래 내용을 스스로 설명해본다.

- 이번 수정 사항을 내 말로 설명할 수 있는가
- 기존 기능에는 어떤 영향을 주는가
- 처음 사용하는 패턴이 있다면 이게 왜 필요하고 왜 좋은가?

처음에는 번거롭게 느껴졌지만 이걸 안하고 넘어갔다가 무슨 수정이 있었는지 기억도 못하는 경우가 종종 있었어서 반드시 하고 넘어가는 팀 컨벤션으로 정했다.

특히 처음 보는 패턴이나 이전에 없었던 케이스가 나오면 그냥 넘어가지 않고 AI에게 왜 이 방식을 썼는지 다시 물어보고 이해가 될 때까지 설명을 듣는다. 이 과정을 거치고 나면 다음에 비슷한 상황이 왔을 때는 AI 없이도 판단할 수 있게 된다.

### 머릿속 규칙을 레포에 커밋하기

가장 중요한건 머릿속에만 규칙을 두면 누락되거나 안하고 넘어가기 쉬우니 프로젝트 내에 규칙 파일을 넣어두고 AI가 참조하게 하는 것이었다. 그래야 하나 이상의 AI나 세션에서도 공통 규칙을 적용해 일관적인 코드 컨벤션이나 패턴, 디자인을 유지할 수 있었다.

나는 이걸 `AGENTS.md`라는 파일로 만들어 커밋했다. 이 저장소에서 작업하는 모든 AI 도구가 착수 전에 읽어야 하는 공통 계약이며 체크 리스트라고 할 수 있다.

## AI는 추론 모델이지 진실 원천이 아니다

물론 처음부터 이렇게 활용했던 것은 아니다. 예전에 TODO로 남겨두었던 `any` 타입을 제거하기 위해 AI에게 리팩터링을 요청한 적이 있었다. any를 구체적인 타입으로 변경했고 빌드도 정상적으로 통과했다.

하지만 해당 응답을 주는 API는 특정 상황에서는 null을 반환할 수 있었는데 이를 고려하지 않은 채 타입을 단정하면서 런타임에서 `Cannot read properties of null (reading 'map')` 오류가 발생했다.

실제 API는 nullable 응답도 허용하고 있었지만 코드만으로는 확인할 수 없었다.

이 경험 이후부터는 타입을 수정할 때도

- 실제 API 응답 형태는 무엇인지
- nullable 값은 없는지
- 기존 호출부와 계약이 깨지지 않는지

를 먼저 확인하는 습관이 생겼다.

## AI에겐 도메인 지식이 없다

AI를 사용하면서 느낀 점은 AI는 프로젝트 밖의 일반적인 지식은 잘 알고 있지만 우리 서비스만의 도메인 맥락까지 알고 있는 것은 아니라는 것이다. 아무리 고맥락 모델이라고 하더라도 코드의 의도나 설계 원칙까지는 AI가 완전히 파악하기 어려워하는 면이 있다고 느꼈다. (아직까지는)

그래서 AI의 답을 그대로 따르기보다는 다른 관점에서 질문하고, 놓친 케이스를 찾고, 설계를 다시 검토하는 데 더 많이 활용한다. 특히 요구사항 검토, 테스트 시나리오 리뷰, 영향 범위 점검, 설계 누락 찾기 등 처럼 혼자서는 놓치기 쉬운 부분에서 가장 큰 도움을 받고 있다.

사수가 없는 환경에서 중요한 건 모든 의사결정의 답을 알고 있는 사람이 되는 것이 아니라 내가 내린 판단을 다시 확인할 수 있는 구조를 만드는 일이라고 생각한다.
